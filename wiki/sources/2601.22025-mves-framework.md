## 概述

MVES（Daniel Commey, 2026年1月）提出 Minimum Viable Evaluation Suite，一个分层的 LLM 应用评测框架。核心发现：**generic "improved" prompt 可能反而降低性能**——task-specific prompt 替换 generic rules 后，抽取准确率从 100% 降至 90%，RAG compliance 从 93.3% 降至 80%。提供了评测方法综合、质量维度分类、LLM-as-Judge 失败模式分析和实用 checklist。

**论文信息**: arXiv:2601.22025, 2026年1月

## 核心贡献

### 1. Define-Test-Diagnose-Fix 评测循环

1. **Define**: 将隐式质量要求转化为可测试的具体属性
2. **Test**: 在 golden set 上评估
3. **Diagnose**: 按失败类别分析，而非只看聚合指标
4. **Fix**: 根据诊断调整 prompt/retrieval/model
5. 循环迭代

### 2. 质量维度分类 (Quality Taxonomy)

| 维度 | 定义 | 适用场景优先级 |
|------|------|---------------|
| Correctness | 事实准确、逻辑合理 | Medical Q&A: High |
| Helpfulness | 实际帮助用户达成目标 | Customer support: High |
| Harmlessness | 避免有害输出 | All: Varies |
| Groundedness | 可追溯到可靠来源 | RAG: High |
| Format adherence | 符合结构要求 | Code gen: High |
| Consistency | 跨查询一致性 | Conversational: High |
| Refusal correctness | 正确拒绝不当请求 | Safety: High |

### 3. MVES 三层框架

- **MVES-Core**: 50-200 case golden set + automated assertions + semantic metric，适用所有应用
- **MVES-RAG**: + retrieval quality (Recall@k, MRR) + faithfulness/grounding checks
- **MVES-Agentic**: + trajectory evaluation + per-tool success rate + sandboxed execution

### 4. 评测方法对比 (Table 2)

| 方法 | 人类相关性 | Cost/1k | 时间 | 回归检测 |
|------|----------|---------|------|---------|
| Human eval | 1.0 (baseline) | $1000+ | Days | High |
| LLM-as-Judge | 0.70-0.85 | $10-50 | Hours | Medium |
| BERTScore | 0.40-0.60 | $0.10 | Mins | Low |
| Exact match | N/A | $0.01 | Secs | High (specific) |

### 5. LLM-as-Judge 失败模式

- **Position bias**: 偏好列表中特定位置的回答
- **Verbosity bias**: 偏好更长的回答
- **Self-preference**: 偏好自己生成的回答
- **Style bias**: 偏好特定写作风格
- **Instruction leakage**: judge prompt 泄漏到评分中

### 6. 实验：Generic prompt 可能有害

| 条件 | Extraction Accuracy | RAG Compliance | Instruction Following |
|------|--------------------|-----------------|-----------------------|
| Task-specific | 100% | 93.3% | 80% |
| + Generic "helpful" wrapper | **90%** (-10%) | **80%** (-13%) | **93%** (+13%) |

Generic rules 与 task-specific constraints 冲突，导致某些任务性能下降。

## 对我们研究的启示

1. **Task-specific > Generic**: 强烈支持我们的 specification-driven 方向——通用的评测 prompt 不如任务特定的规范
2. **Define-Test-Diagnose-Fix 循环**: 为我们的 rubric 迭代提供了系统化框架
3. **Golden set 设计**: 50-200 case 的 golden set 是基础，我们的 11×50=550 case 规模合适
4. **失败模式清单**: 在设计 judge prompt 时需要系统性地防范这些 bias
5. **Quality taxonomy**: 可直接映射到我们的医疗评测维度——Correctness (解决方案)、Helpfulness (追问能力)、Harmlessness (分诊安全性)
6. **MVES-Core 作为最低标准**: 我们的评测系统至少要满足 MVES-Core 的要求
